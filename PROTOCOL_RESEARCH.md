## Protocol Architecture

MTGO uses a **single persistent TLS TCP connection** to port 7770
(`69.174.204.165`). All traffic ‚Äî login, lobby, chat, matchmaking, and live game
actions ‚Äî flows over this one connection.

Game messages are delivered in a two-level envelope:

```
[FLS envelope]  GsMessageMessage (opCode 1153)
                  ‚îî‚îÄ‚îÄ MetaMessage: byte[]  ‚Üê complete CSMessage buffer
                        ‚îî‚îÄ‚îÄ [inner game message] e.g. GamePlayStatusMessage (opCode 4652)
```

The inner MetaMessage is itself a full CSMessage with the standard 8-byte header
and the same binary field serialization. There is no separate game server TCP
connection visible to the client; everything is routed through the FLS.

### Core DLLs

| DLL | Role |
|---|---|
| `Message.dll` | Base `Message` / `CSMessage` classes; framing and pack/unpack primitives |
| `MTGOMessage.dll` | All 1482 concrete message types; the full protocol vocabulary |
| `FlsClient.dll` | Network layer: TLS socket, connection, message pump, opcode sniffer |
| `MTGOEnumStruct.dll` | 553 shared enums and structs; includes `Opcode_t` |
| `WotC.MtGO.Client.Model.Play.dll` | Client-side game state; parses `StateBuf` into `ThingElement` objects |

## Wire Format

### Level 1 ‚Äî Message Frame

Every message on the wire (and every embedded MetaMessage) has this layout:

```
Offset  Size  Type    Description
------  ----  ------  -----------
0       4     int32   Total buffer length in bytes (including this header)
4       2     uint16  OpCode ‚Äî identifies the message type
6       2     uint16  TypeCheckValue ‚Äî first 2 bytes of MD5 of type layout
8+      var   bytes   Payload ‚Äî fields packed sequentially, no padding
```

To read a message from a stream:
1. Read 4 bytes ‚Üí `totalLen` (little-endian int32)
2. Read `totalLen - 4` more bytes
3. Complete message is now in hand; bytes [4-5] are the opCode

### Payload Field Encoding

Fields are written sequentially with no alignment padding:

| .NET type | Wire encoding |
|---|---|
| `bool` | 1 byte (0 or 1) |
| `byte` | 1 byte |
| `ushort` | 2 bytes, little-endian |
| `int` / `uint` | 4 bytes, little-endian |
| `long` / `ulong` | 8 bytes, little-endian |
| `Guid` | 16 bytes, standard GUID byte layout |
| `string` (wide) | `int32` char count + chars as UTF-16 LE (2 bytes/char) |
| `string` (narrow) | `int32` char count + chars as ISO-8859-1 (1 byte/char) |
| nullable value | `bool` has-value byte, then value if present |
| variable-length array | `int32` element count, then each element |

Wide vs. narrow per field is hardcoded in each message class. Wide strings are
used for usernames, deck names, session IDs. Narrow strings for passwords,
fingerprints, and internal tokens.

## Level 3 ‚Äî StateBuf (inside GamePlayStatusMessage)

StateBuf is assembled and parsed by `GamePlayStatusMessageDetailed`
(`WotC.MtGO.Client.Model.Play.dll`). It may arrive split across multiple
`GamePlayStatusMessage` packets and may optionally be diff-compressed.

### Assembly & Diff Algorithm

`Flags` (1 byte in `GamePlayStatusMessage`) controls reassembly:

| Bit | Constant | Meaning |
|---|---|---|
| 0 | `GamestateContainsDiffs` | StateBuf is a diff against the previous full state |
| 1 | `GamestateHead` | Start a new assembly buffer (first/only chunk) |
| 2 | `GamestateTail` | Last chunk ‚Äî process now |

When `Tail` is set but `Head` is not, StateBuf chunks are concatenated in
`game.GameStateAssemblyBuffer`. When `Tail` is set, the buffer is finalised and
then processed (or diffed first if bit 0 is also set).

**Diff format** (`ApplyDiffs2`): a sequence of variable-length opcodes operating
on a cursor over the previous state buffer:

| Leading byte | Meaning |
|---|---|
| `0x00` | Copy: read `uint16` count + `int16` seek-from-current; seek old-state cursor, copy count bytes |
| `0x80` with low 7 bits == 0, next byte == 0 | Long literal: read 3-byte little-endian count, then that many literal bytes |
| `0x80` with low 7 bits == 0, next byte != 0 | Medium copy: count = next byte; `int16` seek; copy from old state |
| `0x80..0xFF` (low 7 bits != 0) | Short copy: count = low 7 bits; `sbyte` seek; copy from old state |
| `0x01..0x7F` | Literal: count = byte value (max 127); read that many literal bytes |

**Checksum** (applied before and after diff): rolling sum seeded at `826366246`,
`checksum = (checksum << 1) + byte` for each byte.

### State Element Container Format

After assembly (and optional diff application), the buffer contains
`NStateElems` elements packed consecutively:

```
Offset  Size  Type            Description
------  ----  --------------  -----------
0       4     int32           Total element size in bytes (including this 8-byte header)
4       4     StateElementType  Element type (enum, see below)
8+      N-8   byte[]          Element payload (N = total size field above)
```

### Element Payload Layouts

#### ThingElement (type 5) ‚Äî game objects (cards, permanents, tokens, spells on stack)

**Payload layout** (decompiled from `ThingElement.ReadBuffer()` and `StateElement.ReadAttributes()`):

```
[int32 from_zone]          ‚Üê Previous zone the card moved from (CardZone enum)
[attribute_list]           ‚Üê PropertyContainer (key-value pairs, terminated by 0x00000000)
```

**Attribute list format** (one entry per property):

```
[uint32 key_with_type]     ‚Üê 0x00000000 terminates the list
[value]                    ‚Üê encoding depends on top 5 bits of key_with_type
```

The `uint32 key_with_type` field encodes both the type and the `MagicProperty` key:
- **Bits 31‚Äì27** (mask `0xF8000000`): attribute type tag
- **Bits 26‚Äì0** (mask `0x07FFFFFF`): `MagicProperty` key (look up in `MagicProperty` enum)

**Type tags and value encodings:**

| Type bits | Tag name | Value encoding |
|---|---|---|
| `0x20000000` | Integer | `int8` (signed byte) |
| `0x28000000` | LargeInt | `int32`; key is remapped: strip `0x08000000`, set `0x20000000` to get the Integer-space key |
| `0x40000000` | String | `uint16 length` then `length` ISO-8859-1 bytes; if `length == 0xFFFF`, instead read `uint32 stringTableIndex` |
| `0x08000000` | List | Recursively parse another attribute list (same format, terminated by `0x00000000`) |
| `0x10000000` | Function | No value (skip; used for function-typed properties, ignored by client) |
| `0x48000000` | StringConstant | (defined but not handled in switch ‚Äî raises error) |

**Go pseudocode for `ReadAttributes`:**

```go
func readAttributes(r *bytes.Reader) map[uint32]any {
    props := map[uint32]any{}
    for {
        var keyWithType uint32
        binary.Read(r, binary.LittleEndian, &keyWithType)
        if keyWithType == 0 { break }

        typeTag := keyWithType & 0xF8000000
        rawKey  := keyWithType & 0x07FFFFFF

        switch typeTag {
        case 0x20000000: // Integer
            var v int8
            binary.Read(r, binary.LittleEndian, &v)
            props[rawKey] = int(v)

        case 0x28000000: // LargeInt ‚Üí remap key to Integer space
            var v int32
            binary.Read(r, binary.LittleEndian, &v)
            remappedKey := (keyWithType & 0xD7FFFFFF) | 0x20000000
            props[remappedKey & 0x07FFFFFF] = int(v)

        case 0x40000000: // String
            var length uint16
            binary.Read(r, binary.LittleEndian, &length)
            if length == 0xFFFF {
                var idx uint32
                binary.Read(r, binary.LittleEndian, &idx)
                props[rawKey] = stringTable[idx]
            } else {
                buf := make([]byte, length)
                r.Read(buf)
                props[rawKey] = string(buf) // ISO-8859-1
            }

        case 0x08000000: // List (nested PropertyContainer)
            props[rawKey] = readAttributes(r)

        case 0x10000000: // Function ‚Äî no value
        }
    }
    return props
}
```

## Login Sequence (for context)

1. Client sends `FlsUserLoginReqMessage` (opCode 1) with:
   - Username (wide string, max 30 chars)
   - Password (narrow string, max 32 chars)
   - DeviceFingerprint (narrow, max 64)
   - ChallengeResponse (narrow, max 6) ‚Äî 2FA if enabled
   - RememberDevice (bool)
   - md5sum = `"696a768998b076f0dec8f6edc8e3652b"` (build hash, hardcoded)
   - ClientFlags (int) ‚Äî bitmask; `ClientFlagEnum.Shiny` for the modern client
   - SessionID (wide string) ‚Äî empty on first login

2. Server responds with session information
3. Client connects to game with `GsConnectMessage` (opCode 1024)

## Opcode Table Tool

**Location**: `tools/opcode-dump/`

Loads `MTGOMessage.dll` via reflection, instantiates every non-abstract
`CSMessage` subclass, reads its `.OpCode`, and outputs a sorted table.

```bash
# Run from tools/opcode-dump directory
dotnet run -c Release -- "<MTGO install dir>" --tsv > opcode-table.tsv
# Auto-discovers MTGO dir if no argument given
dotnet run -c Release -- --tsv > opcode-table.tsv
```

Output file: `opcode-table.tsv` in repo root (1482 entries as of build
3.4.155.4652).

Re-run after every MTGO client update to detect protocol changes.

## Next Steps

### 1. Card Name Coverage (highest priority)

Only 25% of cards have names. The `CARDNAME_STRING` property is not sent for
cards present in the initial state snapshot. Two approaches:

**Option A ‚Äî CARDTEXTURE_NUMBER mapping** (most promising):
Every ThingElement has a `CARDTEXTURE_NUMBER` (card art ID). Build a mapping from
texture IDs to card names/Scryfall IDs. Source: MTGO data files or empirical
correlation from named cards in captures.

**Option B ‚Äî String table decoding:**
Cards with `length == 0xFFFF` names are string table references (`<strtable:N>`).
Decompile the string table population logic from the MTGO client.

### 2. Multi-Game Separation

The decoder currently concatenates all games in a session. Add game boundary
detection to split into separate replay files or add boundary markers.

### 3. Replay Header Population

Capture game_id, player names, and result before the GameOver reset clears state.
Requires:
- `GsPlayerOrderMessage` (opcode 1155) wire layout ‚Äî player names/seats
- `GameResultsMessage` (opcode 4485) wire layout ‚Äî winner determination

### 4. TLS Decryption

Traffic is TLS-encrypted so raw captures are unreadable. Current approach uses
a .NET hook (`tools/capture-hook/`) that patches `FlsClient.Sockets.SslSocketWrapper`
to dump decrypted bytes.

Alternative: `SSLKEYLOGFILE` environment variable (may not work with .NET SslStream).
